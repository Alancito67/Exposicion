Arquitectura de Espacio
en Bases de Datos

Asignación de cuotas · Límites · Tablespaces · Motores

1. ¿Por qué limitamos?

supervivencia del sistema

El problema: Si un usuario guarda archivos o "datos basura" sin control, llena el disco → la base de datos se apaga para todos.

La solución: Asignación de Cuotas → un "presupuesto" de megabytes por persona.

Concepto clave: No es por ser "malos", es por supervivencia del sistema. El equilibrio permite que toda la organización funcione sin caídas.

"Un usuario descontrolado puede hundir un servidor compartido. Las cuotas garantizan estabilidad colectiva."

2. Límites Blandos vs Duros

Límite Blando (Soft Limit)

Como la reserva de gasolina del coche: el sistema te avisa: “Oye, ya casi te acabas tus 10GB, borra algo”.
Te deja seguir, pero ya estás avisado.

Límite Duro (Hard Limit)

Es una pared de concreto. Si llegas al tope, la BD te dice “Error” y no te deja guardar ni una letra más hasta liberar espacio.

En producción se combinan ambos: avisos tempranos + barrera final para proteger la estabilidad.

3. El Tablespace

contenedor físico

Para que la gente entienda: Un Tablespace es como un archivero físico

La base de datos no guarda todo en una sola bolsa. Divide el disco en "archiveros". A un usuario le decimos: “Tú solo puedes usar el cajón número 3, y ese cajón solo mide 1 metro de profundidad”.

Los DBA asignan tablespaces específicos para controlar crecimiento, rendimiento y backup granular.

4. ¿A quién le damos espacio?

Usuarios VIP: Administradores → sin límite (supervisión confiable).
Usuarios de Aplicación: Apps controladas → cuotas ajustadas y monitorizadas.
Usuarios de Prueba: Cuotas muy pequeñas → evitan desastres mientras aprenden.

Estrategia recomendada: segmentar roles y asignar cuotas proporcionales al riesgo/necesidad. Los usuarios de prueba con hard limit bajo evitan accidentes.

5. Diferencias entre Motores

Oracle: El más estricto. Si no le das cuota, el usuario no puede ni crear una tabla. Control granular por tablespace y usuario.
MySQL / MariaDB: Más relajados. Normalmente dejan usar todo el disco a menos que el ingeniero configure límites externos (como quotas del sistema de archivos o plugins).
PostgreSQL: Se enfoca más en permisos de carpetas y límites a nivel de tablespace, pero no tiene cuotas de usuario directas nativas; se gestiona con volúmenes del SO o esquemas separados.
Consejo: En entornos multi-tenant, combina límites del motor + monitoreo proactivo.

Analogía rápida

Restaurante comunitario: Cada comensal (usuario) tiene un plato (cuota). Si alguien llena su plato con comida de sobra y no come, bloquea la mesa. Los límites duros evitan que alguien acapare todo el buffet. ¡El sistema se mantiene vivo para todos!

Conclusión: Las cuotas no son un castigo, sino un mecanismo de convivencia en bases de datos compartidas.

Buenas prácticas

  • Definir soft limit al 85% y hard limit al 100% de la cuota.
  • Auditar usuarios inactivos que acumulan basura.
  • Usar tablespaces separados por entorno (dev, test, prod).
  • Alertas automáticas ante acercamiento al límite duro.
"Un diseño consciente del espacio evita disaster recovery innecesarios."
